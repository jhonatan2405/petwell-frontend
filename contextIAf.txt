  6. AUTENTICACIГ“N
в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
Mecanismo: JWT guardado en localStorage con clave 'petwell_token'

Helpers en utils/auth.ts:
  saveToken(token)       вҶ’ Guardar JWT
  getToken()             вҶ’ Obtener JWT (null si no existe)
  removeToken()          вҶ’ Eliminar JWT (logout)
  isAuthenticated()      вҶ’ boolean (token presente y no vacГӯo)

AuthContext (context/AuthContext.tsx):
  Provee: { user, token, isAuthenticated, loading, login, logout }
  Login: saveToken + actualiza estado React
  Logout: removeToken + limpia estado React

Navbar:
  Lee user.role desde useAuthContext() (no mГЎs isAuthenticated() de localStorage)
  Muestra: "Mi Panel", "ContraseГұa" solo si autenticado
  Muestra: "Panel ClГӯnica" SOLO si user.role === 'CLINIC_ADMIN'
  Oculto para DUENO_MASCOTA y demГЎs roles en desktop y menГә mГіvil
  BotГіn "Cerrar sesiГіn" llama a logout() del contexto + removeToken()

ProtecciГіn de rutas:
  /dashboard         вҶ’ Redirige a /login si no hay token
  /clinic-dashboard  вҶ’ Requiere token + rol CLINIC_ADMIN
                       Si no hay token вҶ’ redirige a /login
                       Si rol incorrecto вҶ’ redirige a /dashboard

RoleGuard (components/auth/RoleGuard.tsx):
  Props: allowedRoles, redirectTo (default: /dashboard), children
  Flujo:
    1. loading вҶ’ spinner
    2. Sin token вҶ’ router.replace('/login')
    3. Rol no permitido вҶ’ router.replace(redirectTo)
    4. Rol correcto вҶ’ renderiza children
  Uso: envuelto en layout.tsx de la ruta protegida

### Regla de sincronizaciуn

Cuando una sesiуn de telemedicina finaliza, el appointment asociado DEBE
marcarse como COMPLETED de forma atуmica desde el backend.

### Implementaciуn (v21.4)

La sincronizaciуn ocurre en el Telemed Service:
  archivo: telemed-service/src/services/telemed.service.ts ? endSession()

Flujo:
  1. PATCH /telemed/sessions/:id/end  (llamada del frontend)
  2. updateSessionStatus(id, { status: 'COMPLETED', ended_at, duration_minutes })
  3. axios.patch(APPOINTMENT_SERVICE_URL/appointments/:appointment_id/status, { status: 'COMPLETED' })
     - fire-and-catch: si falla no revierte la sesiуn, solo loguea el error

### Protecciуn adicional

startSession() ahora bloquea reactivar sesiones COMPLETED o CANCELLED:
  ? Lanza 409 "La consulta ya fue finalizada"

### Frontend (limpio desde v21.4)

  - NO existe lуgica de auto-healing en el frontend
  - handleVetEnd llama endSession() y actualiza UI local con el resultado
  - AppointmentCard.handleEndSession igual: solo llama endSession() sin parches

v21.5 — fix definitivo: sincronizaciуn telemed_session ? appointment al finalizar consulta

CAUSA RAНZ del bug anterior:
  La llamada HTTP del telemed-service al appointment-service usaba el JWT del VETERINARIO.
  El appointment-service valida que appointment.clinic_id === user.clinic_id.
  En algunos escenarios esa validaciуn generaba 403 silencioso (fire-and-catch), dejando
  appointment.status en CONFIRMED aunque telemed_session.status fuera COMPLETED.

SOLUCIУN (patchAppointmentCompleted):
  telemed-service/src/repositories/telemed.repository.ts ? nueva funciуn patchAppointmentCompleted()
  - Actualiza appointments.status = COMPLETED DIRECTAMENTE en Supabase
  - Usa service_role_key ? bypassa RLS y restricciones JWT de rol
  - Sin HTTP, sin JWT, sin restricciones de clinic_id

CAMBIO EN endSession() (telemed.service.ts):
  ANTES: axios.patch ? Appointment Service HTTP (fire-and-catch, puede fallar silencioso)
  AHORA: patchAppointmentCompleted() ? Supabase directo (strict, lanza si falla)

LOGS DE DIAGNУSTICO AСADIDOS:

GARANTНA POST-FIX:
  telemed_session.status = COMPLETED  ?
